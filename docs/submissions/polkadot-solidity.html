tagline
Agent reputation that travels across EVM chains. Now on PolkaVM.
shortDescription
Dispatch deploys ERC-8004 agent identity and reputation contracts on PolkaVM. Workers register once, build verifiable track records per job, and carry that reputation to any EVM chain. Same Solidity, portable trust.
whatItDoes
Dispatch is a compute marketplace where AI agents buy inference from idle hardware. Agents send HTTP requests with x402 payment headers. Workers process jobs (real LLM inference), sign ed25519 receipts, and earn tokens. No accounts, no API keys. The trust problem: how does an agent know which worker to pick? Dispatch uses ERC-8004, an onchain standard for agent identity and reputation. Workers mint agent NFTs, and every completed job posts a reputation score onchain. Higher reputation means priority routing. Today these contracts run on Monad testnet. But ERC-8004 is standard Solidity. It deploys identically on any EVM chain. PolkaVM gives Dispatch a third deployment target and a key unlock: reputation portability across the Polkadot ecosystem. A worker who builds a track record on Monad can have that same identity recognized on PolkaVM. Same contract addresses, same ABI, same trust data. The reputation travels with the worker, not the chain.
polkadotIntegration
Two ERC-8004 Solidity contracts deploy on PolkaVM with zero modifications: Identity Registry (0x8004A818BFB912233c491871b3d84c89A494BD9e on Monad): Workers mint agent NFTs linking to a registration file with capabilities, endpoint URL, and x402 support. Agents discover workers without a centralized directory. Standard ERC-721, fully compatible with PolkaVM's EVM runtime. Reputation Registry (0x8004B663056A597Dffe9eCcC1965A193B7388713 on Monad): After each job, the coordinator posts feedback onchain: score, skill tag, endpoint, feedback hash. Workers accumulate per-skill reputation. Agents query this before routing. Why PolkaVM matters: - Cross-chain portability. A worker on Monad is recognized on PolkaVM with the same agentId. Contracts are byte-for-byte identical. - Polkadot ecosystem access. Parachains query the Reputation Registry to evaluate workers before delegating compute. - Low fees at scale. Per-job reputation updates need cheap onchain writes. PolkaVM delivers this.
howItWorks
Technical flow with PolkaVM deployment: 1. Deploy ERC-8004 contracts to PolkaVM. Standard solc, no Polkadot-specific modifications. Deploy via Hardhat or Foundry targeting PolkaVM's EVM endpoint. 2. Worker calls registerAgent() to mint an agent NFT on PolkaVM. Sets agentURI pointing to a JSON file with capabilities (skills, endpoint, x402 support). 3. Worker connects to the coordinator via WebSocket. Coordinator reads reputation from whichever chain has the richest history (Monad, PolkaVM, or both). 4. Agent submits a job via HTTP POST with x402 payment header. Coordinator scores workers: ERC-8004 reputation (0-100) x stake multiplier (1x/1.5x/2x). Best score wins. 5. Worker runs the job. LLM inference via Ollama (desktop) or text processing (mobile). Signs an ed25519 receipt over the SHA-256 output hash. 6. Coordinator posts reputation feedback to PolkaVM. Score, skill tag, job type recorded onchain. Cross-chain reputation grows. Same protocol, same contracts, new chain.
