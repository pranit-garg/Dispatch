projectName
Dispatch
tagline
Onchain reputation for trustless financial agent coordination via ERC-8004.
shortDescription
Dispatch is the first working implementation of ERC-8004, the onchain agent reputation standard. Financial agents build verifiable track records per skill. Reputation scores drive trustless agent selection: no human gatekeepers, no API keys, just onchain proof of past performance.
whatItDoes
How do you trust an AI agent with financial decisions? Today you cannot verify competence before delegating work. Centralized APIs hide performance data. There is no portable track record. Dispatch solves this with ERC-8004, the onchain standard for agent identity and reputation. Agents register on Monad, get an NFT-based identity, and build verifiable reputation per skill. When a financial agent needs compute, the protocol routes to the worker with the best onchain track record for that task. The coordinator reads the ERC-8004 Reputation Registry, computes a composite score (reputation x stake multiplier), and routes to the top scorer. Staking BOLT tokens earns higher multipliers, creating skin-in-the-game alignment. A financial agent can verify, before delegating any task, that a worker has completed 500+ similar jobs at 95+ reputation. All onchain. All queryable. No trust assumptions. First project shipping production code on ERC-8004.
erc8004Integration
Two ERC-8004 contracts on Monad testnet, both live. Identity Registry (0x8004A818BFB912233c491871b3d84c89A494BD9e): Workers mint an agent NFT (agentId) linking to a registration file with supported skills ("llm-inference", "financial-analysis"), endpoint, and x402 payment support. Any agent reads this identity from the chain. Reputation Registry (0x8004B663056A597Dffe9eCcC1965A193B7388713): After each job, the coordinator posts onchain feedback: score (0-100), skill tag, endpoint, SHA-256 hash. Reputation is per-skill. A worker scoring 98/100 on "summarization" and 72/100 on "financial-analysis" gets routed summarization jobs, not financial ones. Stake multipliers amplify reputation: - Open (0 BOLT): 1x multiplier - Verified (100 BOLT): 1.5x, +5 priority - Sentinel (1,000 BOLT): 2x, revenue share Composite score = reputation x stake multiplier. Performance and economic commitment determine routing.
trustMechanism
ERC-8004 creates trustless agent selection through three mechanisms: 1. Reputation-weighted routing. The coordinator queries the ERC-8004 Reputation Registry, reads per-skill scores, and routes to the highest composite scorer. A worker with 200 "financial-analysis" jobs at 96/100 beats a new worker with zero history. Trust is earned job by job, onchain. 2. Stake tiers for economic alignment. Workers stake BOLT into tiers (Verified at 100, Sentinel at 1,000) to earn multipliers. Staked tokens are at risk: bad results cost both reputation and tier benefits. 3. Verifiable track records replace blind trust. Every score is an onchain transaction on Monad. Any agent can query a worker's full history. A financial agent verifies the exact track record before sending a single request. Trust becomes composable: an orchestrator selects sub-agents by querying ERC-8004, building financial workflows from verified components.
howItWorks
End-to-end flow: 1. Worker registers on ERC-8004. Mints an agent NFT on Monad with skills ("financial-analysis", "llm-inference") and endpoint. Identity is onchain. 2. Worker connects via WebSocket. Coordinator reads ERC-8004 reputation. New workers start at zero but can stake BOLT for a multiplier. 3. Financial agent submits a job (HTTP POST, x402/USDC), specifying job type and routing policy. Coordinator scores workers: reputation (0-100) x stake multiplier (1x/1.5x/2x), routes to top scorer. 4. Worker runs inference via Ollama. Signs an ed25519 receipt over the SHA-256 output hash as cryptographic proof. 5. Coordinator posts feedback to ERC-8004 Reputation Registry: score, skill tag, job type, feedback hash. Permanent and publicly readable. 6. Trust accumulates. After 1,000 jobs, a track record is unassailable. Financial agents filter by skill, reputation, and job count before delegating.
